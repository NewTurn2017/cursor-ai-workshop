# 실습 3: 작은 추가 + 짧은 수정 (디버깅 & 검토)

> **4단계 중**: 구현 + 검토
> **소요 시간**: 약 10분
> **사용 기능**: Chat (Cmd+L), Inline Edit (Cmd+K), Agent (Cmd+I)

## 목표

- 구현 뒤에 붙는 **검토를 습관**으로 만든다.
- "원인 먼저" 패턴으로 디버깅하는 법을 익힌다.

## Step 1: 작은 기능 추가

실습 2에서 만든 결과물에 작은 요구를 추가합니다:

```text
FAQ 개수를 5로 바꿔도 3개만 나오는 문제를 고쳐줘.
```

또는 기능 추가:

```text
공지문에 "꼭 읽어야 할 핵심 안내" 박스를 추가해줘.
```

## Step 2: 디버깅 템플릿 사용

문제가 있을 때 아래 형식으로 요청합니다:

```text
증상:
- FAQ 개수를 5로 바꿔도 3개만 보여

재현:
- FAQ 개수를 5로 바꾼 직후 생성 결과 확인

원하는 답변:
1) 원인을 먼저 설명하고
2) 고쳐주고
3) 확인 방법을 적어줘
```

## Step 3: 검토 문장 붙이기

수정이 끝나면 반드시:

```text
바뀐 점과 확인 방법을 알려줘.
```

## 최소 안전 규칙

- 실제 고객 정보, 비밀번호, 키는 넣지 않는다
- 큰 변경은 이유를 먼저 설명하게 한다
- 파괴적 명령은 범위를 먼저 설명하게 한다
- 변경 뒤에는 확인 방법을 반드시 남긴다
- 불확실하면 가정을 적게 한다

## 모델 선택 기준

| 상황 | 권장 |
|------|------|
| 일상 작업, 반복 작업 | Auto (기본값) |
| 구조 설계, 어려운 디버깅 | 강한 모델 선택 |

핵심은 모델보다 **범위**입니다. 범위가 작아지면 품질과 비용이 같이 좋아집니다.

## 성공 기준

- [ ] "원인 먼저" 패턴으로 디버깅 요청했다
- [ ] 수정 후 확인 방법을 받았다
- [ ] diff를 확인했다
- [ ] 검토 문장을 마지막에 붙였다
