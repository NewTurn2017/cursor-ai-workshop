# Cursor AI 3시간 실습 교육 커리큘럼

> **교육일**: 2026년 4월 1일 (화)  
> **소요시간**: 3시간 (180분)  
> **대상**: Cursor를 처음 쓰거나, AI 작업 흐름을 더 또렷하게 잡고 싶은 사람  
> **핵심 주제**: **나만의 바이브코딩 4단계 — 리서치 → 계획 → 구현 → 검토**

---

## 이번 워크숍의 한 줄

- AI를 잘 쓰는 사람은 기능을 많이 아는 사람이 아니라 **4단계로 일을 굴리는 사람**이다.
- 기능은 주제가 아니라 **4단계를 돕는 도구**다.
- 좋은 결과는 **짧은 계획 + 작은 범위 + 마지막 검토**에서 나온다.

## 오늘 바로 쓰는 4문장

1. **리서치** — “먼저 상황을 정리해줘”
2. **계획** — “시작 전에 5줄 계획을 보여줘”
3. **구현** — “이 범위만 만들어줘”
4. **검토** — “바뀐 점과 확인 방법을 알려줘”

## 운영 방식

### 기본 루프

이 워크숍은 설명보다 **짧은 반복**으로 진행한다.

1. **리서치** — 지금 상황과 필요한 입력을 먼저 본다.
2. **계획** — 5줄 계획으로 범위를 고정한다.
3. **구현** — 한 번에 크게 말고, 한 범위씩 만든다.
4. **검토** — 바뀐 점과 확인 방법을 마지막에 남긴다.

### 실습 원칙

- 실습은 전부 **mock 데이터 기반**으로 진행한다.
- 한 대화에는 **한 기능 또는 한 수정**만 담는다.
- 막히면 대화를 늘리기보다 **계획으로 돌아간다**.
- 결과를 받으면 바로 넘기지 말고 **검토 문장 1개**를 붙인다.

## M1. 기준 잡기 + 세팅 완료 (25분)

### 목표

- 2026년 Cursor를 기능 목록이 아니라 **작업 흐름 도구**로 이해한다.
- 실습에 필요한 기본 단축키와 설정을 끝낸다.

### 핵심 메시지

- Cursor 학습의 중심은 기능 이름이 아니라 **일을 나누는 순서**다.
- 리서치는 Chat과 탐색 질문으로, 계획은 Plan Mode로, 구현은 Tab/Inline/Agent로, 검토는 diff와 확인 문장으로 가져간다.

### 설치 체크리스트

1. Cursor 실행 및 로그인
2. 실습 폴더 열기
3. 인덱싱 완료 확인
4. 단축키 확인
   - `Cmd+I` / `Ctrl+I` — Agent
   - `Cmd+L` / `Ctrl+L` — Chat
   - `Cmd+K` / `Ctrl+K` — Inline Edit
   - `Shift+Tab` — Plan Mode
5. Privacy 설정 확인

### 첫 질문

```text
이 프로젝트 구조를 짧게 정리해줘.
```

## M2. 리서치 — 상황과 구조 파악 (30분)

### 목표

- Tab, Chat, Inline Edit를 기능 시연이 아니라 **리서치 도구**로 써본다.
- 짧은 질문 하나로 구조를 파악하는 감각을 익힌다.

### 핵심 메시지

- Tab은 방향을 빠르게 이어주는 도구다.
- Chat은 설명과 요약에 강하다.
- Inline Edit는 범위를 좁힌 수정에 강하다.
- 초반 예제에서도 Plan Mode가 먼저 질문하게 하면 구현이 훨씬 안정된다.

### 실습 1 — 회의 메모 / 프로젝트 업데이트 정리기

**권장 프롬프트**

```text
회의 메모 / 프로젝트 업데이트 정리기를 만들어줘.
먼저 상황을 정리하고,
내가 결정해야 할 것을 3~5개 질문하고,
답을 반영한 5줄 계획을 보여준 뒤,
그 다음 가벼운 결과 화면을 만들어줘.
- 입력: 회의 메모 또는 프로젝트 업데이트 텍스트
- 출력: 대상별 요약, 다음 액션 3개, 결정사항, 공유용 한 줄 업데이트
- 모바일 대응
- 한 화면에 가볍게
```

**실습 후 바로 붙일 질문**

```text
이 화면 구조와 사용 흐름을 3문장으로 설명해줘.
```

### 코칭 포인트

- 결과가 길어지면 “한 화면 안에 들어오게 줄여줘”를 붙인다.
- 질문이 먼저 나오면 잘 되고 있는 것이라고 안내한다.
- 참가자가 막히면 설명을 대신 길게 해주기보다 “먼저 상황을 정리해줘”를 먼저 쓰게 한다.

## M3. 계획 — 5줄 계획으로 범위 고정 (40분)

### 목표

- Agent와 Plan Mode를 **계획 → 구현** 흐름으로 연결한다.
- 멀티파일 작업에서도 작은 범위를 유지하는 법을 익힌다.

### 핵심 메시지

- 계획은 길수록 좋은 게 아니라 **짧고 선명할수록** 좋다.
- Agent는 바로 던지는 도구가 아니라, **짧은 계획 뒤에 붙이는 도구**다.

### 실습 2 — 공지/FAQ 생성기

**권장 프롬프트**

```text
공지/FAQ 생성기를 만들어줘.
먼저 아래 정보 중 빠진 것을 질문해줘:
- 대상
- 채널
- 톤
- 필수 포함 항목
- FAQ 개수
- 입력 형식

답을 받은 뒤 5줄 계획을 보여주고,
그 다음 가벼운 결과 화면을 만들어줘.
기능: 공지 제목/본문 초안, FAQ 3~5개, 꼭 읽어야 할 핵심 요약, 복사하기 쉬운 출력 블록
```

**2차 프롬프트**

```text
이 생성기를 더 쓰기 좋게 다듬어줘.
- 대상별 톤 전환
- 꼭 읽어야 할 안내 박스 추가
- FAQ 개수 선택 UI 추가
```

### 컨텍스트 운영 원칙

- 한 대화 = 한 기능 또는 한 수정
- 범위가 바뀌면 새 대화
- 이전 내용은 요약만 다시 넣기

## M4. 구현 후 검토 — 확인까지 끝내기 (30분)

### 목표

- 구현 뒤에 붙는 검토를 습관으로 만든다.
- 규칙, 디버깅, 모델 선택을 모두 **검토 관점**으로 묶는다.

### 핵심 메시지

- 검토는 마지막 장식이 아니라 **품질을 확정하는 단계**다.
- AI가 빠를수록 검토 문장은 더 짧고 또렷해야 한다.

### 짧은 디버깅 템플릿

```text
증상:
- FAQ 개수를 5로 바꿔도 3개만 보여

재현:
- FAQ 개수를 5로 바꾼 직후 생성 결과 확인

원하는 답변:
1) 원인을 먼저 설명하고
2) 고쳐주고
3) 확인 방법을 적어줘
```

### 최소 안전 규칙

- 실제 고객 정보, 비밀번호, 키는 넣지 않는다.
- 큰 변경은 이유를 먼저 설명하게 한다.
- 파괴적 명령은 범위를 먼저 설명하게 한다.
- 변경 뒤에는 확인 방법을 반드시 남긴다.
- 불확실하면 가정을 적게 한다.

### 모델 선택 기준

- **Auto**: 기본값
- **가벼운 모델**: 일상 작업, 반복 작업
- **강한 모델**: 구조 설계, 어려운 디버깅, 큰 수정

핵심은 모델보다 **범위**다. 범위가 작아지면 품질과 비용이 같이 좋아진다.

## M5. 4단계 미니 프로젝트 (25분)

### 목표

- 각자 바로 떠올릴 수 있는 작은 도구 하나를 만든다.
- 리서치 → 계획 → 구현 → 검토를 끝까지 한번 돌린다.

### 프로젝트 옵션

1. **회의 메모 정리기**
   - 메모를 요약, 결정사항, 다음 액션으로 정리
2. **공지/FAQ 생성기**
   - 입력값으로 공지문과 FAQ 초안 생성
3. **고객 문의 분류기**
   - 문의를 유형, 우선순위, 답변 초안으로 분류
4. **주간 업데이트 카드**
   - 한 주 진행상황을 공유용 카드 한 장으로 정리

### 진행 방식

1. 리서치 문장 1개 쓰기
2. 계획 5줄 받기
3. 한 범위 구현하기
4. 검토 문장으로 마무리하기
